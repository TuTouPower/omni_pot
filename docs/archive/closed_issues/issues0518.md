## 问题背景与根因分析

### 背景

这批问题不是单个控件的小 bug，而是一次“UI 对齐 example”“功能对齐 example”和“测试按真实用户路径验收”之间的断层。当前实现里，很多地方已经有了界面骨架、配置项或测试入口，但没有按真实用户路径闭环验证：双击单文件首次启动、托盘菜单、设置/识别窗口加载、设置窗口尺寸、全局快捷键、OCR/TTS 默认服务、下拉框弹层、i18n 文案、主题和透明背景等都需要按最终产品行为重新收敛。

同时，review 指出其中一部分不是“代码明显写错”，而是**用户反馈 / example / `docs/spec.md` / 现有测试之间互相冲突**。这类问题必须先把规格改清楚，否则直接实现会继续造成文档、测试和代码互相打架。

### 已确认的主要根因

1. **首次启动行为与欢迎页要求存在规格缺口**
   - 当前代码启动时始终先创建翻译窗口，再在 `isFirstRun()` 时创建配置窗口。
   - 当前测试把“首次运行打开配置窗口”当作正确行为。
   - 当前 `docs/spec.md` 未明确定义独立欢迎页，只提到翻译窗口空状态引导。
   - 结论：这是 example/用户期望与当前规格冲突，不应直接写成已确认实现 bug。

2. **窗口打开转圈是通用 lazy-loading 路径问题**
   - `src/App.tsx` 把翻译、设置、截图、识别、词典、更新器窗口都定义为 `React.lazy(...)`。
   - 所有窗口共用 `<Suspense fallback={<Spinner />}>`。
   - 因此设置页和识别窗口先显示加载旋转，不能只按设置页单点优化；需要测量各窗口的 chunk 加载、配置读取、服务注册和页面初始化耗时。

3. **托盘菜单模型曾只有 5 个基础项（已修复）**
   - 修复前 `electron/tray/index.ts` 只定义并渲染输入翻译、剪贴板监听、配置、重启、退出。
   - 已补齐 OCR 识别、截图翻译入口，并接入截图识别/截图翻译动作。
   - 已由 i18n / tray E2E 断言中英文菜单项和对应动作。

4. **设置窗口主进程尺寸和渲染根节点尺寸曾不一致（已修复）**
   - 修复前主进程创建设置窗口为 `800x600`，而 `src/windows/config/index.tsx` 的 `.op-window` 内联尺寸为 `880x600`。
   - 已改为配置窗口创建宽度与设置页布局匹配，设置页根节点跟随视口尺寸，E2E 断言设置页不再出现横向溢出。

5. **设置窗口曾没有复用可拖动标题栏模式（已修复）**
   - 可拖动区域依赖 `-webkit-app-region: drag`。
   - 修复前设置窗口头部是普通 `div`，没有 drag region，所以设置窗口不能拖动。
   - 已给设置窗口顶部区域添加 drag region，并由 E2E 断言标题栏为 `drag`、置顶/关闭按钮为 `no-drag`。

6. **设置页下拉框弹层曾被卡片裁剪（已修复）**
   - `ConfigSelect` 原本把下拉层作为 `.select` 内部的 `position: absolute` 子节点。
   - `.card` 全局设置了 `overflow: hidden`，因此任何放在卡片内的设置下拉框都可能被裁剪，翻译语言下拉框只是最明显的表现。
   - 已改为共享 `ConfigSelect` 使用 portal + fixed 弹层，并由 E2E 断言下拉选项能真实接收指针点击。

7. **设置页语言名曾走当前界面语言（已修复）**
   - `src/i18n/language_names.ts` 的普通 `language_options(t, ...)` 会调用当前 i18n 翻译生成语言名。
   - 设置页已改用 native language options，翻译设置和识别设置中的语言项显示 `English`、`日本語` 等语言自身文字。
   - `auto` 没有语言自身名称，仍按当前界面语言显示为“自动检测”或 `Auto Detect`。

8. **快捷键测试仍缺少完整真实触发链路**
   - `src/windows/config/hotkey_settings.tsx` 能调用 `hotkey:register` 并保存配置。
   - 已补充重复快捷键冲突测试，防止后绑定功能偷走先绑定功能的 accelerator。
   - 已补充输入翻译已注册 hotkey action 测试，防止可见翻译窗口被热键隐藏而不是进入输入状态。
   - 已补充划词翻译/划词字典选中文本成功路径和无选中文本反馈测试，防止快捷键静默失败。
   - 已补充截图识别/截图翻译已注册 hotkey action 测试，防止快捷键未打开截图覆盖层。
   - 已补充系统注册失败路径测试，防止系统拒绝注册时 UI 错误保存配置。
   - 输入翻译、划词翻译、划词字典的真实系统热键路径仍需 Windows smoke 验证。

9. **OCR/TTS 服务注册存在，但默认列表为空且当前规格也写为空**
   - OCR/TTS registry 里有服务注册逻辑。
   - `DEFAULT_CONFIG.recognize_service_list` 和 `DEFAULT_CONFIG.tts_service_list` 是空数组。
   - `docs/spec.md` 和服务管理 E2E 也把 OCR/TTS 默认空列表写成期望。
   - 结论：如果要默认显示系统 OCR / Tesseract / TTS 服务，需要先更新规格和测试，再改默认配置。

10. **检查更新按钮曾使用缺失的 i18n key（已修复）**
   - About 页使用 `about.check_update`。
   - 修复前 locale 里该 key 不存在，只有 `general.check_update`。
   - 已补齐 `about.check_update` 中英文 locale，并移除中文 fallback；E2E 已断言按钮显示本地化文本且不显示 raw key。

11. **服务管理 UI 曾同时存在拖动排序和上下移动按钮（已修复）**
    - 服务项已经接入 dnd-kit 拖动排序。
    - 修复前仍保留 move up / move down 按钮，导致界面出现不需要的额外排序控件。
    - 已移除箭头式排序控件并放大编辑/删除按钮；E2E 已断言服务项不再出现上下移动按钮。

12. **透明背景、主题/主色等属于规格对齐问题**
    - 开发者模式已按用户反馈从当前规格、配置类型、默认配置、设置 UI 和 E2E 期望中移除。
    - Wayland 提示已从快捷键页默认 UI 和当前规格中移除；如需说明，只应作为故障排查文档或错误态展示。
    - 透明背景入口已在 Windows 等平台显示；新建窗口会读取 `transparent` 配置。
    - 当前 spec 明确为 2 种主题 + 5 种主色；设置页已补齐 5 个主色按钮，并将主色广播到已打开窗口。

13. **`D/Program Files` 异常目录来自 Git Bash 的 MSYS 路径转换**
    - 触发命令是在 Git Bash 中执行 Windows 的 `wsl.exe`，同时把 `/home/karon/karson_ubuntu/github_repo/...` 作为参数传入。
    - MSYS 会在调用 Windows 可执行文件前自动把类 Unix 路径转换为 Windows 路径，实际传给 WSL 的 clone 目标变成 `D:/Program Files/Git/home/karon/karson_ubuntu/github_repo/chinese-dictionary`。
    - WSL 内部把这个字符串当作普通 Linux 相对路径；回到 Windows 文件系统时，文件名中的冒号 `:` 不能直接表示，于是显示为私用字符 ``，最终在项目目录出现 `D/Program Files/...`。
    - 避免方式：Git Bash 调 `wsl.exe` 时加 `MSYS_NO_PATHCONV=1`，或使用 `wsl -d Ubuntu-22.04 bash -c "mkdir -p /home/... && cd /home/... && git clone ..."`，让 WSL 路径只在 WSL shell 内解释。

## 解决方案候选

### 方案 A：逐条修补

按当前 18 个开放问题逐条改代码、逐条补测试。

- 优点：单项变更小，容易快速看到局部进展。
- 缺点：这些问题共享窗口模型、设置控件、服务默认值、测试策略等根因，逐条修容易互相打架，也容易继续漏掉真实用户路径。

### 方案 B：先建立验收矩阵，再按模块修复（推荐）

先把 example 对齐项转成验收矩阵：启动、托盘、设置窗口、快捷键、OCR/TTS、服务管理、主题/主色、透明背景、i18n。每个模块先补失败测试或人工验收脚本，再做最小实现修复。

- 优点：能防止“只修 UI 不修功能”，也能防止修完一个问题又漏掉同类问题。
- 缺点：开始阶段会比直接改代码慢一点，因为要先补验收标准和规格决策。

### 方案 C：先整体重做设置页和托盘体验

把设置窗口、服务管理、快捷键、主题等作为一次 UI/UX 重构来处理。

- 优点：最终一致性最好。
- 缺点：范围过大，容易引入新问题，不符合当前“只动必须动的”原则。

**推荐采用方案 B**：先把真实用户路径变成可验证标准，再按模块最小修复。当前问题数量多，但根因集中，适合用验收矩阵控制范围。

## 建议并行 sub agent 分工

以下工作相互依赖较少，适合开多个 sub agent 并行推进；每个 agent 都必须先读本文件、`docs/spec.md`、相关测试和 example，不得只看单个代码文件。

1. **规格与验收矩阵 agent**
   - 范围：阶段 0 和阶段 1。
   - 任务：把 18 个问题映射到确认缺陷 / 规格冲突 / 需运行验证；同步 `docs/spec.md`、`PLAN.md` 和测试验收目标；产出最终 checklist。
   - 不应改功能代码。

2. **设置窗口与共享控件 agent**
   - 范围：设置窗口尺寸、拖动区域、`ConfigSelect` 弹层裁剪、图标尺寸。
   - 任务：修共享窗口/控件层，避免每个设置页分别打补丁。
   - 需要重点跑设置页 E2E 和浏览器人工验收。

3. **托盘与快捷键 agent**
   - 范围：托盘菜单、全局快捷键注册后触发真实功能、无选中文本时可见反馈、外部 API 非 stub。
   - 任务：贯通主进程、托盘、hotkey、selection、screenshot、translate/dict/recognize 窗口。
   - 需要 Windows 实机或打包 smoke 验证。

4. **OCR/TTS 与服务管理 agent**
   - 范围：默认服务策略、服务列表、服务添加/编辑/删除、服务排序 UI、识别窗口服务选择和失败反馈。
   - 任务：按阶段 0 决策修默认配置、服务页和 OCR/TTS 入口。
   - 需要覆盖 config service E2E、recognize E2E 和实际 OCR/TTS 可用性。

5. **i18n / 主题 / 透明背景 agent**
   - 范围：语言自身名称、`about.check_update`、raw key 扫描、主题/主色、透明背景入口和行为。
   - 任务：修用户可见文案和外观设置，避免只改中文或只改单页。
   - 需要多语言 E2E 和视觉验收。

6. **打包与人工 smoke agent**
   - 范围：`npm run dist`、单文件 exe 首次启动、托盘、快捷键、截图、设置/识别窗口真实体验。
   - 任务：验证自动测试覆盖不到的 Windows 桌面行为，记录不可自动化的人工验收结果。
   - 不应直接实现功能修复，只反馈阻塞和复现步骤。

7. **测试重审与失败用例 agent**
   - 范围：所有当前通过但守护错误行为的 Playwright/Vitest 测试。
   - 任务：按 example 和真实用户路径重审测试目标，先改出失败用例或人工验收项，再交给对应功能 agent 修实现。
   - 重点：不能只检查 `data-testid` 或配置值，要覆盖用户可见结果、窗口状态、菜单项、快捷键触发、OCR/TTS 默认行为和 i18n 文案。
