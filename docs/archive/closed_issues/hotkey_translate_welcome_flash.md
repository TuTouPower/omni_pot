# 翻译热键先闪欢迎页问题分析

## 1. 文档目的

本文给后续接手 Omni Pot 的开发者说明一个具体运行时问题：用户按下翻译快捷键后，翻译窗口会先闪一下欢迎页，然后才进入翻译内容页。

本文不是修复记录，也不是最终规格，而是问题分析文档。它需要让没有参与前序讨论的人看懂：

- Omni Pot 是什么项目。
- 相关功能在产品里承担什么职责。
- 当前技术框架和进程边界是什么。
- 这个问题为什么出现。
- 为什么不能简单回退。
- 可选修复方案有哪些。
- 最推荐的修复方向是什么。

## 2. 项目背景

Omni Pot 是一个跨平台桌面翻译、文字识别和词典工具。项目目标是用新的技术栈重写并延续 Pot Desktop 的核心能力，同时补齐现代桌面应用需要的测试、配置、服务管理和发布流程。

产品形态是常驻系统托盘的桌面应用。用户通常不会像普通网页应用一样一直停留在主窗口，而是通过全局快捷键、托盘菜单、剪贴板监听、截图入口等方式临时唤起不同窗口。

项目的主要用户场景包括：

- 选中一段文本后按快捷键翻译。
- 没有选中文本时按快捷键打开输入翻译窗口。
- 截图后进行文字识别或截图翻译。
- 查询单词或短语的词典释义。
- 使用多个翻译服务同时返回结果。
- 使用本地或在线服务进行语音朗读、语言检测、二维码识别等辅助能力。

因此，Omni Pot 的交互重点不是“打开一个固定主页面”，而是“从系统任意位置快速唤起正确的小窗口，并尽快进入正确状态”。这也是本问题重要的原因：快捷键路径的首屏速度和首屏内容都直接影响体感。

## 4. 相关产品功能

本问题涉及的是翻译窗口和翻译快捷键。

翻译快捷键 `hotkey_translate` 是一个合并入口。它不是单纯的“输入翻译”，也不是单纯的“选中翻译”。它的行为取决于用户当前是否选中了文本：

| 用户状态 | 快捷键行为 |
|---|---|
| 当前系统焦点处有选中文本 | 读取选中文本，打开翻译窗口，填入源文本并开始翻译 |
| 当前没有选中文本 | 打开翻译窗口，显示输入框，等待用户手动输入 |

这个合并入口的设计目的是减少快捷键数量，让用户只记一个“翻译”快捷键。代价是主进程需要在快捷键触发后先判断有没有选中文本。

翻译窗口还有一个欢迎页。欢迎页用于普通空窗口状态，展示欢迎信息和快捷键提示。它适合“用户主动打开空翻译窗口、还没有输入内容”的场景。

但欢迎页不适合“热键刚触发，系统正在读取选区”的场景。此时窗口内容为空只是暂时的，不代表用户真的打开了一个空翻译窗口。

## 5. 相关运行链路

### 5.1 主进程热键入口

全局快捷键注册和触发逻辑在 `electron/hotkey/index.ts`。

`hotkey_translate` 触发后会进入 `triggerTranslateEntry`。这个函数做三件事：

1. 读取当前选中文本。
2. 创建或聚焦翻译窗口。
3. 根据选区结果向翻译窗口发送 IPC。

当前代码的关键行为是：读取选区和创建窗口并行进行。

也就是说，热键触发后，主进程不会等选区读取完成才开窗口，而是马上开窗口，然后等选区结果回来再发消息。

### 5.2 选中文本读取

选中文本读取入口在 `electron/selection/index.ts`。

Windows 上主要读取链路在 `electron/selection/windows.ts` 和 `electron/selection/clipboard.ts`：

1. 优先通过 UI Automation 读取当前焦点元素的文本选区。
2. 如果 UI Automation 读取不到，再模拟 `Ctrl+C`。
3. 写入剪贴板 sentinel，等待剪贴板内容变化。
4. 读取到文本则返回文本。
5. 超时或失败则返回空结果。
6. 最后恢复用户原剪贴板内容。

这个流程为了尽量不破坏用户剪贴板，必须做备份、写 sentinel、轮询、恢复。它天然比普通函数调用慢。

`electron/selection/clipboard.ts` 中剪贴板轮询默认最长等待约 300ms。UI Automation 路径本身也可能因为目标应用复杂而耗时。也就是说，“读取选区”不是一个可以忽略成本的同步小操作。

### 5.3 窗口管理

窗口管理在 `electron/windows/manager.ts`。

翻译窗口通过 `focusOrCreate(WindowLabel.TRANSLATE, get_translate_window_options())` 创建或聚焦。窗口加载完成前，主进程可以通过 `sendWhenReady` 排队发送 IPC。等渲染进程发出 `renderer:ready` 后，队列中的消息再发送给窗口。

这个机制允许“先开窗口、后发文本”。最近的性能修复正是利用了这个机制。

### 5.4 渲染进程翻译窗口

翻译窗口 UI 在 `src/windows/translate/index.tsx`。

它通过 Zustand 保存源文本、翻译结果、翻译状态、语言状态等。它监听主进程发来的几个关键事件：

| IPC | 含义 |
|---|---|
| `translate:from-selection` | 主进程读到了选中文本，要求填入并翻译 |
| `translate:input-translate` | 主进程没有读到选中文本，要求进入输入模式 |
| `translate:from-api` | HTTP API 触发翻译 |
| `translate:from-clipboard` | 剪贴板监听触发翻译 |

欢迎页显示条件目前大意是：

- 源文本为空。
- 没有强制显示输入框。
- 用户没有跳过欢迎页。
- 当前不是“选区为空”的输入模式。

问题在于：这里没有“正在等待热键选区结果”的状态。

## 6. 历史背景：为什么之前要改

之前的问题是：用户没有选中文本时按翻译热键，窗口弹出很慢。

旧流程类似：

1. 用户按下翻译热键。
2. 主进程等待 `readSelectedText()` 完成。
3. 如果没有选区，读取流程仍然要走 UI Automation 和剪贴板 fallback。
4. 读取失败或超时后，主进程才创建翻译窗口。
5. 用户看到窗口。

这个设计的好处是窗口出现时状态准确；坏处是窗口出现太晚。尤其是无选区路径，本来用户只是想打开输入框，却必须先等一轮选区读取失败。

为了解决这个问题，最近做过一次修复：

- 应用启动时预加载平台选区 reader，避免第一次热键触发时动态加载模块。
- `triggerTranslateEntry` 中先启动 `readSelectedText()`，不等待它完成。
- 立即创建或聚焦翻译窗口。
- 等选区结果回来后再发送 `translate:from-selection` 或 `translate:input-translate`。

相关归档文档：`docs/archive/closed_issues/hotkey_translate_waits_for_selection.md`。

这次修复解决了慢弹窗，但引入了本文件讨论的欢迎页闪烁问题。

## 7. 当前问题

现象：用户选中文本后按翻译热键，翻译窗口不是直接进入翻译状态，而是先短暂显示欢迎页，然后才显示源文本和翻译结果。

用户看到的是：

1. 按快捷键。
2. 翻译窗口出现。
3. 欢迎页闪一下。
4. 源文本出现。
5. 翻译结果开始加载或显示。

这不是期望行为。用户已经选中了文本，热键语义是“翻译这段文本”，不应该出现欢迎页。欢迎页只适合真正的空窗口。

## 8. 根因分析

根因是状态建模不完整。

当前实现把“源文本为空”直接推导成“可以显示欢迎页”。但在热键并行开窗后，源文本为空至少有三种完全不同的含义：

| 状态 | 源文本为空的原因 | 应该显示欢迎页吗 |
|---|---|---|
| 普通打开翻译窗口 | 用户确实还没有输入内容 | 应该 |
| 热键刚触发 | 主进程正在读取选区，文本尚未送达 | 不应该 |
| 热键读取完成且无选区 | 用户想进入输入翻译 | 不应该，应该显示输入框 |

最近的性能修复增加了“窗口已显示但选区结果尚未返回”的时间窗口，但前端没有对应状态，所以错误地套用了普通空窗口逻辑。

更具体的链路：

1. `triggerTranslateEntry` 启动 `readSelectedText()`。
2. `triggerTranslateEntry` 立即调用 `focusOrCreate(WindowLabel.TRANSLATE)`。
3. 翻译窗口加载 React。
4. 这时 `translate:from-selection` 还没到。
5. Zustand 中 `sourceText` 仍为空。
6. `show_welcome_empty` 被计算为 true。
7. 欢迎页被渲染。
8. 选区读取完成。
9. 主进程发送 `translate:from-selection`。
10. 前端设置 `sourceText`。
11. 欢迎页消失。

所以，这不是渲染性能问题，也不是欢迎页组件的问题，而是缺少“热键读取中”这个中间状态。

## 9. 为什么不能简单回退

最直接的修法是回退到旧流程：先等选区读取完成，再开窗口。

但这样会恢复原来的问题：空按热键弹窗慢。

这个项目的翻译热键必须兼顾两类用户动作：

- 有选区：尽快把选区送进窗口翻译。
- 无选区：尽快打开输入框。

旧流程牺牲了无选区速度。新流程保住了速度，但首屏状态不完整。正确方向不是回退，而是在新流程里补齐状态。

## 10. 可选方案

### 10.1 方案 A：新增“热键选区读取中”状态

热键触发后，主进程立即打开翻译窗口，并立刻告诉翻译窗口：当前是热键触发，正在等待选区读取结果。

前端收到这个状态后：

- 不显示欢迎页。
- 可以显示极轻的等待态、空白占位或输入区域骨架。
- 等最终结果到达后再切换到翻译态或输入态。

状态转换：

| 事件 | 前端状态 |
|---|---|
| 收到 `translate:selection-pending` | `selectionPending = true`，禁止欢迎页 |
| 收到 `translate:from-selection` | `selectionPending = false`，填入文本并翻译 |
| 收到 `translate:input-translate` | `selectionPending = false`，清空文本并显示输入框 |

优点：

- 保留当前快弹窗收益。
- 直接修正根因。
- 不污染用户配置。
- 不影响普通打开翻译窗口时的欢迎页。
- 状态边界清楚，适合写测试。

缺点：

- 需要新增一个 IPC 或等价的窗口启动上下文。
- 前端增加一个短暂状态。
- 需要补测试，避免后续重构时再次把空状态混在一起。

### 10.2 方案 B：窗口先隐藏，等选区读取完成后再显示

主进程仍然创建窗口，但先不显示。等 `readSelectedText()` 返回后，再根据结果显示正确界面。

优点：

- 用户不会看到欢迎页闪烁。
- 前端改动少。

缺点：

- 有选区和无选区都要等读取完成后才看到窗口。
- 空按热键会重新变慢。
- 只是隐藏错误首屏，没有补齐状态模型。
- 冷启动时还要考虑窗口隐藏加载和最终显示时机，复杂度不一定真的低。

### 10.3 方案 C：回退并行开窗改动

恢复旧逻辑：`await readSelectedText()` 后再 `focusOrCreate()`。

优点：

- 实现最简单。
- 不会出现欢迎页闪烁。

缺点：

- 原来的慢弹窗问题复发。
- 与最近修复目标相反。
- 用户空按热键时体感差。

### 10.4 方案 D：热键路径禁用欢迎页

只要是热键打开翻译窗口，就不显示欢迎页。

优点：

- 改动可能比方案 A 小。
- 可以消除闪烁。

缺点：

- 语义粗糙。
- 如果热键打开后最终确实没有选区，也需要进入输入模式，而不是简单“不显示欢迎页”。
- 如果实现成写入 `welcome_dismissed`，会污染用户长期配置。
- 容易影响普通空窗口欢迎页。

### 10.5 方案 E：缩短选区读取耗时

优化 UI Automation 或剪贴板 fallback，让选区读取更快，减少闪烁时间。

可能方向：

- 缩短剪贴板轮询超时。
- 给 UI Automation 增加软超时。
- 把 UI Automation 移到 worker 或 utility process。

优点：

- 可以改善整体热键响应。
- 对其他选区相关功能也可能有帮助。

缺点：

- 不能从根上解决状态错误；只要存在异步窗口，仍可能闪。
- 缩短超时可能降低选区读取成功率。
- worker 化 UI Automation 改动大，COM 初始化和资源释放要谨慎。

## 11. 推荐方案

推荐方案 A：新增“热键选区读取中”状态。

核心判断：本问题不应该用“隐藏窗口”或“回退性能优化”解决。当前真正缺失的是状态表达。只要把热键读取中的中间状态显式表达出来，就可以同时满足：

- 窗口仍然快速出现。
- 有选中文本时不闪欢迎页。
- 无选中文本时仍能快速进入输入模式。
- 普通打开空翻译窗口时继续显示欢迎页。

推荐的最终状态表：

| 场景 | 触发来源 | 源文本 | 中间状态 | 应显示 |
|---|---|---|---|---|
| 普通打开翻译窗口 | 托盘 / 开发默认窗口 / 手动打开 | 空 | 无 | 欢迎页 |
| 热键刚触发 | `hotkey_translate` | 空 | 选区读取中 | 等待态或空白占位 |
| 热键读取到选区 | `hotkey_translate` | 有文本 | 无 | 源文本 + 翻译结果 |
| 热键未读取到选区 | `hotkey_translate` | 空 | 无 | 输入框 |
| API 翻译 | HTTP API | 有文本 | 无 | 源文本 + 翻译结果 |
| 剪贴板触发翻译 | 剪贴板监听 | 有文本 | 无 | 源文本 + 翻译结果 |

## 12. 推荐实现方向

建议改动边界保持最小，只围绕热键翻译入口和翻译窗口状态。

### 12.1 主进程

在 `triggerTranslateEntry` 中保留当前并行流程：

1. 先创建 `result_promise`。
2. 立即 `focusOrCreate(WindowLabel.TRANSLATE, get_translate_window_options())`。
3. 立刻发送一个新的 pending 事件，例如 `translate:selection-pending`。
4. 等 `result_promise` 完成。
5. 有文本则发送 `translate:from-selection`。
6. 无文本则发送 `translate:input-translate`。

注意点：

- pending 事件必须走 `sendWhenReady`，否则窗口冷启动时可能丢消息。
- pending 事件只应该用于热键翻译入口，不应该影响 API 翻译、剪贴板翻译、普通打开窗口。
- 如果翻译窗口已经存在，也应该发送 pending 事件，让旧内容或欢迎页进入正确等待态。

### 12.2 preload / IPC 类型

需要在 preload 暴露新的监听函数，例如：

- `onTranslateSelectionPending(callback)`

共享 IPC 类型中也应补齐这个事件，避免主进程和渲染进程事件名散落。

### 12.3 渲染进程

在 `src/windows/translate/index.tsx` 增加本地状态，例如：

- `selection_pending`

状态规则：

- 收到 `translate:selection-pending`：设为 true，取消已有延迟翻译，清理 selection empty 状态，禁止欢迎页。
- 收到 `translate:from-selection`：设为 false，写入文本并翻译。
- 收到 `translate:input-translate`：设为 false，进入输入模式。
- 用户手动输入、API 翻译、剪贴板翻译开始时：确保设为 false。

欢迎页条件需要改成：

- 源文本为空。
- 非强制输入模式。
- 未跳过欢迎页。
- 非选区为空输入态。
- 非热键选区读取中。

也就是：热键读取中永远不能显示欢迎页。

### 12.4 等待态 UI

等待态不需要复杂设计。推荐优先用最小 UI：

- 保留标题栏。
- 主体区域可以为空白，或显示很轻的“正在读取选中文本...”提示。
- 不要显示欢迎页。
- 不要提前显示错误。

如果担心等待态文字也会闪，可以先用空白占位。因为这个状态通常只持续几十到几百毫秒，最重要的是不要显示错误语义的欢迎页。

## 13. 测试建议

需要补测试防止回归。

### 13.1 单元测试

在热键单元测试中保留现有断言：窗口必须在选区 promise resolve 前创建。

新增断言：

- `triggerTranslateEntry` 在选区 promise resolve 前发送 `translate:selection-pending`。
- pending 事件发生在最终 `translate:from-selection` 或 `translate:input-translate` 之前。

### 13.2 渲染进程测试

如果已有渲染层测试能力，应覆盖：

- 收到 pending 后，`welcome-empty` 不出现。
- 收到 from-selection 后，pending 结束，源文本显示。
- 收到 input-translate 后，pending 结束，输入框显示。

### 13.3 E2E 测试

建议加用户路径测试：

1. 启动应用，确保 `welcome_dismissed=false`。
2. 关闭或清空翻译窗口。
3. 模拟热键触发，选区读取 promise 延迟返回文本。
4. 断言窗口已经出现，但欢迎页没有出现。
5. 选区文本到达后，断言源文本等于选区文本。

另一路：

1. 模拟热键触发，选区读取返回空。
2. 断言窗口进入输入模式。
3. 断言欢迎页没有先闪，也没有最终显示。

## 14. 验收标准

修复后应满足：

- 有选中文本按翻译热键：不出现欢迎页闪烁，直接进入翻译流程。
- 没有选中文本按翻译热键：窗口快速出现，最终显示输入框。
- 普通打开空翻译窗口：欢迎页仍正常显示。
- 用户点击欢迎页跳过：`welcome_dismissed` 行为不变。
- `hotkey_translate` 弹窗速度不能回退到旧的“等待选区读取完成后才显示窗口”。
- 已有翻译、API 翻译、剪贴板翻译、词典热键不受影响。

## 15. 不推荐做法

不要把 `welcome_dismissed` 当作热键状态使用。它是用户偏好，不是一次热键触发的临时状态。

不要为了消除闪烁而回退到先等 `readSelectedText()` 再开窗。这样会恢复旧性能问题。

不要只缩短剪贴板超时来掩盖问题。选区读取仍然是异步的，只要窗口先显示，就必须有正确的中间状态。

不要让普通空翻译窗口失去欢迎页。欢迎页本身是合理的，只是不应该出现在热键读取选区的中间阶段。

## 16. 结论

这次 bug 是一次典型的性能优化引入状态缺口：为了解决快捷键空按弹窗慢，主进程改成并行读取选区和创建窗口；窗口变快了，但渲染进程仍按旧模型理解空状态，于是把“正在等待选区结果”误判成“普通空窗口”，显示了欢迎页。

最推荐修法是新增“热键选区读取中”状态，让前端明确知道当前空文本只是临时状态。这样既保留快弹窗，又避免欢迎页闪烁，并且不会污染用户配置或破坏普通欢迎页逻辑。
