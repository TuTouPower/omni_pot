# DeepL 免费 JSON-RPC 长文本限制

> 更新日期：2026-05-18
> 适用范围：Omni Pot 当前 `deeplx_free` / DeepL 免费 JSON-RPC 路径。

本文记录 DeepL 免费 JSON-RPC 接口在长文本输入下返回原文或截断英文的真实请求探测结果。结论仅针对当前环境和当前接口行为；DeepL 上游可能随时调整策略。

---

## 结论

DeepL 免费 JSON-RPC 接口不是“多段文本一定失败”，也不是“有换行一定失败”。目前观察到的触发条件更接近：

> **单个 DeepL job / sentence 中放入较长英文文本时，接口会不稳定；在约 900～1000 英文字符附近开始出现返回原文或截断英文原文。**

已确认：

1. **空行不是充分条件**：短多段文本 `Hello.\n\nGoodbye.` 可正常翻译成中文。
2. **换行不是必要条件**：去掉所有换行后的约 960 字符长文本也会异常返回原文。
3. **长度是主要触发因素**：单段 890 字符仍能翻译；单段 1068 字符会异常返回英文/截断英文。
4. **阈值不是精确固定值**：931、960、967、1068 字符均出现异常，但是否触发可能受内容、标点、重复度、DeepL 内部检测和限流策略影响。
5. **原 `1.txt` 失败的根因不是 case number 本身**：更可能是整篇多段长文本被作为一个 DeepL job 发送，单 job 文本过长触发了免费接口异常路径。

因此，当前实现采用参考项目 `pot-desktop` 的更高层 JSON-RPC 路径：

> **使用 `LMT_handle_texts`，并设置 `splitting: 'newlines'`，让 DeepL 服务端自行切分文本，而不是本地构造单个 `LMT_handle_jobs` job。**

## 为什么“多段文本”看起来像触发条件

原问题中，`1.txt` 是一段多段英文客服申诉文本，包含：

- 多个段落；
- case number / plan / issue / request 等结构化行；
- 较长正文段落；
- 总长度约 1000 字符。

旧实现把整篇文本作为一个 `jobs[0].sentences[0].text` 发送给 DeepL 免费 JSON-RPC。这样会让多段文本整体落入“单 job 长文本”场景，因此表现为“多段文本返回原文”。

但真实探测表明：

- 短多段文本可以正常翻译；
- 长单段文本也可以失败；
- 长文本即使去掉换行也可以失败。

所以“多段”只是常见外观，不是根本条件。

## 对 Omni Pot 的实现

DeepL 免费模式不能把用户输入整段塞进一个 `LMT_handle_jobs` job。当前实现已切换为：

```json
{
  "method": "LMT_handle_texts",
  "params": {
    "splitting": "newlines",
    "texts": [
      { "text": "...", "requestAlternatives": 3 }
    ]
  }
}
```

该格式来自参考项目 `pot-desktop`，由 DeepL 服务端按 `newlines` 策略处理文本切分，并返回 `result.texts[0].text`。

当前真实请求探测中，免费 JSON-RPC 的 `LMT_handle_texts` 对 `PT-BR` / `PT-PT` 这类葡萄牙语区域码返回 HTTP 400；因此 `deeplx_free` 会在该路径下退回基础语言码 `PT`，避免 `pt_br` / `pt_pt` 配置直接失败。

如果后续 `LMT_handle_texts` 仍出现长文本异常，再考虑在本地增加保守分片策略：优先按段落拆，再按句子或空格拆，确保每个请求片段控制在 **700～800 英文字符以内**。

## 测试建议

外部服务问题应尽量使用真实请求验证，不要只靠 mock 隐藏上游行为。

当前 opt-in 外部服务测试覆盖：

1. 真实 DeepL 请求：普通短文本应返回非空结果。
2. 真实 DeepL 请求：多段长文本不应返回原文。
3. 真实 DeepL 请求：单段超长文本不应返回英文原文或截断英文。
4. 如需 mock，仅用于难以稳定复现的异常结构，例如 DeepL 少返回某个 job 的响应。

真实请求测试应使用合成、非敏感文本，不能把用户真实 case number、账号、支持邮件内容发送给第三方服务。

---

## 风险

DeepL 免费 JSON-RPC 不是官方公开 API：

- 上游可能限流；
- 请求头或 JSON-RPC 格式可能失效；
- 长文本阈值可能变化；
- 返回原文不一定带错误码；
- 真实服务测试可能因网络环境短暂失败。

因此，DeepL 免费模式适合作为无 key 默认翻译服务之一，但不应被当作稳定官方 SLA 服务。需要高稳定性时，应使用 DeepL official API 或其他可配置翻译服务。
