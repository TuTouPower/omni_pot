## 问题背景与根因分析

### 背景

这批问题不是单个控件的小 bug，而是一次“UI 对齐 example”“功能对齐 example”和“测试按真实用户路径验收”之间的断层。当前实现里，很多地方已经有了界面骨架、配置项或测试入口，但没有按真实用户路径闭环验证：双击单文件首次启动、托盘菜单、设置/识别窗口加载、设置窗口尺寸、全局快捷键、OCR/TTS 默认服务、下拉框弹层、i18n 文案、主题和透明背景等都需要按最终产品行为重新收敛。

同时，review 指出其中一部分不是“代码明显写错”，而是**用户反馈 / example / `docs/spec.md` / 现有测试之间互相冲突**。这类问题必须先把规格改清楚，否则直接实现会继续造成文档、测试和代码互相打架。

### 已确认的主要根因

1. **首次启动行为与欢迎页要求存在规格缺口**
   - 当前代码启动时始终先创建翻译窗口，再在 `isFirstRun()` 时创建配置窗口。
   - 当前测试把“首次运行打开配置窗口”当作正确行为。
   - 当前 `docs/spec.md` 未明确定义独立欢迎页，只提到翻译窗口空状态引导。
   - 结论：这是 example/用户期望与当前规格冲突，不应直接写成已确认实现 bug。

2. **窗口打开转圈是通用 lazy-loading 路径问题**
   - `src/App.tsx` 把翻译、设置、截图、识别、词典、更新器窗口都定义为 `React.lazy(...)`。
   - 所有窗口共用 `<Suspense fallback={<Spinner />}>`。
   - 因此设置页和识别窗口先显示加载旋转，不能只按设置页单点优化；需要测量各窗口的 chunk 加载、配置读取、服务注册和页面初始化耗时。

3. **托盘菜单模型只有 5 个基础项**
   - `electron/tray/index.ts` 只定义并渲染输入翻译、剪贴板监听、配置、重启、退出。
   - OCR 识别、截图翻译等 example 中应有入口没有进入托盘菜单模型。
   - 现有 i18n / tray 测试还把这 5 项写成期望值，所以测试会守护错误的不完整菜单。

4. **设置窗口主进程尺寸和渲染根节点尺寸不一致**
   - 主进程创建设置窗口为 `800x600`。
   - `src/windows/config/index.tsx` 的 `.op-window` 内联尺寸为 `880x600`。
   - 这会导致内容初始展示和窗口边界不对应，用户需要手动缩放才能看起来正常。

5. **设置窗口没有复用可拖动标题栏模式**
   - 可拖动区域依赖 `.op-titlebar { -webkit-app-region: drag; }`。
   - 设置窗口头部是普通 `div`，没有 drag region，所以设置窗口不能拖动。

6. **设置页下拉框弹层被卡片裁剪是通用结构问题**
   - `ConfigSelect` 把下拉层作为 `.select` 内部的 `position: absolute` 子节点。
   - `.card` 全局设置了 `overflow: hidden`。
   - 因此任何放在卡片内的设置下拉框都可能被裁剪，翻译语言下拉框只是最明显的表现。

7. **翻译设置语言名走当前界面语言，不是语言自身名称**
   - `src/i18n/language_names.ts` 的 `language_options(t, ...)` 会调用当前 i18n 翻译生成语言名。
   - `shared/types/language.ts` 的 fallback 里 `en` 是“英文”，不是 `English`；`auto` 是“自动检测”，没有语言自身名称，需要单独按产品约定显示。
   - 当前 i18n 测试也按“界面语言变化后语言名跟着变化”来断言。
   - 这与“语言项应显示自身文字，例如 English、日本語、한국어”的需求相反。

8. **快捷键测试只验证注册保存，没有验证真实触发链路**
   - `src/windows/config/hotkey_settings.tsx` 能调用 `hotkey:register` 并保存配置。
   - `tests/user_e2e/specs/config_settings.spec.ts` 只断言“绑定成功”和配置值更新。
   - 没有测试按下全局快捷键后是否真的打开输入翻译、划词翻译、划词字典、截图识别、截图翻译。
   - 对于划词功能，当前读取不到选中文本时只写日志并直接返回，用户没有可见反馈。

9. **OCR/TTS 服务注册存在，但默认列表为空且当前规格也写为空**
   - OCR/TTS registry 里有服务注册逻辑。
   - `DEFAULT_CONFIG.recognize_service_list` 和 `DEFAULT_CONFIG.tts_service_list` 是空数组。
   - `docs/spec.md` 和服务管理 E2E 也把 OCR/TTS 默认空列表写成期望。
   - 结论：如果要默认显示系统 OCR / Tesseract / TTS 服务，需要先更新规格和测试，再改默认配置。

10. **检查更新按钮用了缺失的 i18n key**
   - About 页使用 `about.check_update`。
   - 当前 locale 里该 key 不存在，只有 `general.check_update`。
   - 这可能导致 raw key 或非中文界面中文 fallback 泄漏。
   - 现有测试没有断言检查更新按钮文案，所以漏掉了。

11. **服务管理 UI 同时存在拖动排序和上下移动按钮**
    - 服务项已经接入 dnd-kit 拖动排序。
    - 同时仍保留 move up / move down 按钮，导致界面出现不需要的额外排序控件。

12. **开发者模式、Wayland 提示、透明背景、主题/主色等属于规格对齐问题**
    - 开发者模式仍在 `docs/spec.md`、配置、UI 和测试中存在；用户反馈说已经去掉，需先做产品决策。
    - Wayland 提示当前符合 `docs/spec.md`，但不符合用户对 example 的解释；需决定 example 是否覆盖 spec。
    - 透明背景当前只在 Mac 平台显示入口；用户期望 Windows 也有入口和功能。
    - 当前 spec 是 2 种主题 + 5 种主色，当前实现只有 `system/light/dark`，缺少主色设置入口；“5 种主题”说法需改为“主题/主色设置对齐”。

## 解决方案候选

### 方案 A：逐条修补

按当前 26 个开放问题逐条改代码、逐条补测试。

- 优点：单项变更小，容易快速看到局部进展。
- 缺点：这些问题共享窗口模型、设置控件、服务默认值、测试策略等根因，逐条修容易互相打架，也容易继续漏掉真实用户路径。

### 方案 B：先建立验收矩阵，再按模块修复（推荐）

先把 example 对齐项转成验收矩阵：启动、托盘、设置窗口、快捷键、OCR/TTS、服务管理、主题/主色、透明背景、i18n。每个模块先补失败测试或人工验收脚本，再做最小实现修复。

- 优点：能防止“只修 UI 不修功能”，也能防止修完一个问题又漏掉同类问题。
- 缺点：开始阶段会比直接改代码慢一点，因为要先补验收标准和规格决策。

### 方案 C：先整体重做设置页和托盘体验

把设置窗口、服务管理、快捷键、主题等作为一次 UI/UX 重构来处理。

- 优点：最终一致性最好。
- 缺点：范围过大，容易引入新问题，不符合当前“只动必须动的”原则。

**推荐采用方案 B**：先把真实用户路径变成可验证标准，再按模块最小修复。当前问题数量多，但根因集中，适合用验收矩阵控制范围。

   - 对确认缺陷类问题，直接补失败测试和修复。

### 阶段 1：建立验收基线

目标：先把“测试守护当前实现”的问题修掉，让测试能失败在真实问题上，避免继续出现“测试全绿但用户不可用”。

1. **全量重审并重写问题测试**
   - 逐个审查 `tests/user_e2e/specs/*`、关键 Vitest 用例和 E2E 辅助 API，删除或改写“守护当前错误实现”的断言。
   - 判定标准：与 example/已确认规格冲突的断言是守护错误实现；只验证配置值、`data-testid` 或控件存在但不验证用户可见结果的是覆盖不足；与当前 spec 一致但 spec 待决策的测试先标为阶段 0 后复审。
   - 所有 example 对齐项必须先形成会失败的用户路径测试或人工验收步骤，再进入实现修复。
   - 重点重写：首次启动、托盘菜单、窗口 bounds/拖动/加载、快捷键真实触发、OCR/TTS 默认服务、服务排序 UI、语言显示、下拉框弹层、关于页文案、主题/主色、透明背景。

2. **整理 example 对齐清单**
   - 输出启动、托盘、设置、快捷键、OCR/TTS、服务、主题/主色、窗口行为的验收项。
   - 验证：每个开放问题都能映射到至少一个验收项或一个产品决策。

3. **补真实用户路径测试**
   - 首次启动：验证已确认的首屏行为。
   - 托盘：验证 example 要求的菜单项和对应动作。
   - 设置窗口：验证 bounds、内容展示、拖动区域、下拉框不裁剪。
   - 快捷键：验证绑定后触发实际功能，而不仅是配置保存。
   - OCR/TTS：按确认后的默认策略验证服务列表和识别/朗读入口。
   - i18n：验证用户可见文本不显示 raw key，也不在非中文界面落回中文。

4. **区分自动测试和人工验收**
   - 能自动测的进入 E2E/集成测试。
   - 依赖 Windows 全局快捷键、截图、托盘真实交互、单文件双击的项目，补人工验收步骤或专门 smoke 脚本。

### 阶段 2：窗口与设置控件修复

目标：先解决设置页的基础可用性，因为它影响多个问题。

1. **统一设置窗口尺寸**
   - BrowserWindow 尺寸和渲染根节点尺寸必须一致。
   - 打开后内容不应需要用户缩放才能完整展示。

2. **设置窗口增加可拖动区域**
   - 复用或对齐现有 titlebar drag/no-drag 规则。
   - pin、close、nav、select 等交互区域必须是 no-drag。

3. **修复 ConfigSelect 弹层裁剪**
   - 下拉层不能被 `.card overflow: hidden` 裁剪。
   - 优先使用 portal/fixed 弹层或调整卡片结构，不只针对翻译语言下拉框做局部 hack。

4. **修复图标尺寸规则**
   - 普通符号尺寸与汉字视觉大小一致。
   - 置顶和关闭窗口两个符号可比汉字略大。

### 阶段 3：设置项语义对齐

目标：把设置页从“有控件”变成“控件语义正确且功能可用”。

1. **语言显示改为语言自身名称**
   - 翻译源语言、目标语言、第二语言等显示 `English`、`日本語` 等自身文字。
   - `auto` 没有语言自身名称，需在阶段 0/规格中明确显示为当前界面语言的“自动检测”，还是固定显示 `Auto` / `Auto Detect`。

2. **按阶段 0 决策处理 Wayland 提示和开发者模式**
   - 若 example 覆盖当前 spec，则移除常驻 Wayland 提示并更新 spec。
   - 若开发者模式不再属于产品功能，则同步移除 spec、配置 UI、测试和相关实现入口。

3. **补齐主题/主色和透明背景入口及行为**
   - 对齐 spec/example 中确认后的主题与主色模型。
   - 透明背景不只显示按钮，还要有可验证的窗口行为。

### 阶段 4：服务与 OCR/TTS 修复

目标：让识别、翻译、朗读服务从默认状态就符合确认后的规格。

1. **按阶段 0 决策设置 OCR/TTS 默认服务列表**
   - 如果默认不为空，OCR 至少包含可用的系统 OCR 或 Tesseract 入口。
   - 如果默认不为空，TTS 至少包含可用的内置或在线服务入口。
   - 同步更新 `docs/spec.md`、默认配置和 E2E 期望。

2. **服务页移除多余上下排序按钮**
   - 保留拖动排序。
   - 删除或隐藏箭头式排序控件。

3. **放大服务编辑和删除按钮**
   - 图标和点击区域都要符合设置页整体尺寸规则。

4. **截图识别服务选择闭环**
   - 识别窗口能选择 OCR 服务。
   - 截图识别能使用当前启用服务识别图片。
   - 识别失败时给用户可见反馈。

### 阶段 5：托盘与快捷键功能闭环

目标：所有入口都要触发真实功能，不再只保存配置。

1. **托盘菜单对齐 example**
   - 添加 OCR 识别、截图翻译等缺失项。
   - 每个菜单项都有对应动作和测试/验收。

2. **快捷键触发真实功能**
   - 输入翻译：按下后显示翻译窗口并进入输入状态。
   - 划词翻译：有选中文本时翻译；无选中文本时给用户可见反馈。
   - 划词字典：有选中文本时查词典；无选中文本时给用户可见反馈。
   - 截图识别：截图后打开识别窗口并显示 OCR 结果或错误。
   - 截图翻译：截图后 OCR，再把结果送入翻译窗口。

3. **补全外部 API 行为**
   - `/recognize` 不能只是 stub。
   - 若保留 curl 替代方案，文档中的 API 必须对应真实可用行为。

### 阶段 6：i18n 和可见文案扫尾

目标：用户界面不显示 raw key，不出现 example-only 文案误用。

1. **修复 `about.check_update` 文案**
   - 使用存在的 key，或补齐 about 命名空间翻译。

2. **扫描关键页面 raw key**
   - 设置、关于、服务、快捷键、识别、更新页都要覆盖。

3. **测试断言用户可见文本**
   - 不只通过 test id 找控件，也要断言关键文案。

### 阶段 7：最终验证

1. **自动验证**
   - `npm run typecheck`
   - `npm run lint`
   - `npm test`
   - 关键 E2E：启动、设置、服务、快捷键、OCR、托盘、i18n。

2. **打包验证**
   - `npm run dist`
   - 双击单文件 exe 验证首次启动、托盘、设置窗口、快捷键、OCR/TTS。

3. **人工验收清单**
   - 打开设置页不需要缩放。
   - 设置窗口可拖动。
   - 语言下拉框不被裁剪。
   - 托盘项完整且能触发功能。
   - 快捷键对应功能都有可见结果。
   - OCR/TTS 服务默认策略符合规格。
   - 关于页检查更新不显示 raw key 或错误语言 fallback。

## 建议并行 sub agent 分工

以下工作相互依赖较少，适合开多个 sub agent 并行推进；每个 agent 都必须先读本文件、`docs/spec.md`、相关测试和 example，不得只看单个代码文件。

1. **规格与验收矩阵 agent**
   - 范围：阶段 0 和阶段 1。
   - 任务：把 26 个问题映射到确认缺陷 / 规格冲突 / 需运行验证；同步 `docs/spec.md`、`PLAN.md` 和测试验收目标；产出最终 checklist。
   - 不应改功能代码。

2. **设置窗口与共享控件 agent**
   - 范围：设置窗口尺寸、拖动区域、`ConfigSelect` 弹层裁剪、图标尺寸。
   - 任务：修共享窗口/控件层，避免每个设置页分别打补丁。
   - 需要重点跑设置页 E2E 和浏览器人工验收。

3. **托盘与快捷键 agent**
   - 范围：托盘菜单、全局快捷键注册后触发真实功能、无选中文本时可见反馈、外部 API 非 stub。
   - 任务：贯通主进程、托盘、hotkey、selection、screenshot、translate/dict/recognize 窗口。
   - 需要 Windows 实机或打包 smoke 验证。

4. **OCR/TTS 与服务管理 agent**
   - 范围：默认服务策略、服务列表、服务添加/编辑/删除、服务排序 UI、识别窗口服务选择和失败反馈。
   - 任务：按阶段 0 决策修默认配置、服务页和 OCR/TTS 入口。
   - 需要覆盖 config service E2E、recognize E2E 和实际 OCR/TTS 可用性。

5. **i18n / 主题 / 透明背景 agent**
   - 范围：语言自身名称、`about.check_update`、raw key 扫描、主题/主色、透明背景入口和行为。
   - 任务：修用户可见文案和外观设置，避免只改中文或只改单页。
   - 需要多语言 E2E 和视觉验收。

6. **打包与人工 smoke agent**
   - 范围：`npm run dist`、单文件 exe 首次启动、托盘、快捷键、截图、设置/识别窗口真实体验。
   - 任务：验证自动测试覆盖不到的 Windows 桌面行为，记录不可自动化的人工验收结果。
   - 不应直接实现功能修复，只反馈阻塞和复现步骤。

7. **测试重审与失败用例 agent**
   - 范围：所有当前通过但守护错误行为的 Playwright/Vitest 测试。
   - 任务：按 example 和真实用户路径重审测试目标，先改出失败用例或人工验收项，再交给对应功能 agent 修实现。
   - 重点：不能只检查 `data-testid` 或配置值，要覆盖用户可见结果、窗口状态、菜单项、快捷键触发、OCR/TTS 默认行为和 i18n 文案。
