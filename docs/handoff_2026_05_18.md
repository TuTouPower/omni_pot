## 三、严格遵守的约束（user 强偏好）

来自 [CLAUDE.md](../CLAUDE.md) 和已保存的记忆：

1. **绝对不要改 demo 文件**（`docs/design/omni_pot/*.jsx`, `*.html`, `*.css`）。只能在 [docs/design/demo_todo.md](design/demo_todo.md) 记录差异。
2. **每次代码修改后跑 `npm run dist`**，并在汇报里分开写：自动化结果 / `npm run dist` 结果 / 真实打包产物 smoke 结果。没跑过真实 smoke 不能写"已修复"，只能写"自动化路径通过，packaged 行为未验证"。详见 [docs/test.md §3.3](test.md)。
3. **少 mock，多真实**。免费/无需密钥的服务走真实 API；只有付费/需密钥才 mock。E2E 用 `playwright route` 而非全局桩。
4. **snake_case** 全栈统一：变量、函数、文件名、目录名。例外只有 `CLAUDE.md`、`README.md`。
5. **缩进 4 空格**，禁止 tab。**日志用 logging 模块**（应用代码内），禁止 `console.log` 调试输出（`scripts/` 下例外）。
6. **简洁优先**：只动必须动的；不顺手重构旁边的；不为不可能的场景写错误处理；不写"灵活配置"超出需求。
7. **文档同步**：每次改代码检查 [docs/](.) 和 [CLAUDE.md](../CLAUDE.md) 是否需同步。开放限制写 [docs/issues.md](issues.md) 或 [PLAN.md](../PLAN.md)。
8. **术语**："语音合成"是服务/设置类别名；"朗读"是用户在按钮看到的动作名。同一个 TTS 能力，不要当成两个独立功能。
9. **"配置" → "设置"**：UI 文案统一为"设置"。代码变量名可以保留 `config`（这是 `terminology_settings.spec.ts` 的范围）。

## 六、可能踩的坑

- **新增 testid 时记得同步 Page Object**：[tests/user_e2e/pages/translate_page.ts](../tests/user_e2e/pages/translate_page.ts) 与同目录其它 PO。否则其它 spec 后续要用也得改。
- **改 `translate_pinned` 配置时**：默认值、迁移逻辑（`config.json` 老用户）、序列化反序列化、`config_settings.spec.ts` 里 `expect_config(omni, 'translate_pinned', ...)` 都要补。
- **改托盘 popover** 后，[i18n.spec.ts](../tests/user_e2e/specs/i18n.spec.ts) `expect_tray_labels` 数组里的 7 项要扩成 9 项（中英文都要改）。
- **窗口圆角透明**：Windows 上 `transparent: true` 与 `backgroundColor` 互斥；如果 BrowserWindow 设了 `backgroundColor: '#ffffff'` 必须删掉。
- **e2e 端点**：[electron/server/index.ts](../electron/server/index.ts) 只在 `OMNI_POT_E2E=1` + `OMNI_POT_E2E_TOKEN` 匹配时启用，不影响生产。新增端点同步更新 [e2e_api.ts](../tests/user_e2e/fixtures/e2e_api.ts)。
- **`translate_window_constraints.spec.ts`** 里直接读了 `translate['page']` 这种"私有"访问（`page` 在 Page Object 里是 private）。如果 strict TS 报错，把 `translate_page.ts` 的 `private page` 改成 `readonly page`，或在 PO 暴露专门的 evaluate 方法。
- **CI 不要被 red 卡死**：如果短时间内无法修完全部 9 条，可以临时在 PR-gate（`test:e2e:core`）只跑 `@core` 标签的 spec；新增的 9 条都带 `@ui` 标签，不会卡 PR 门。但 nightly 跑 `full` 一定要红到绿才能合。
